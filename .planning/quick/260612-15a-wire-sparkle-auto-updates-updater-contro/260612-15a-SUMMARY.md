# Phase quick-260612-15a Plan 01: Wire Sparkle Auto-updates Summary

Sparkle 2.9.0 was already linked but inert. This plan makes it functional end-to-end: a single live, app-scoped `SPUStandardUpdaterController` starts at launch with automatic-check scheduling; users get a "Check for Updates…" menu action and a Settings "Updates" section (auto-check toggle + Check Now); the built `Info.plist` now carries the real EdDSA public key (from the login Keychain) and a version-independent GitHub latest-asset feed URL; and `release.sh` generates, verifies, signs, and uploads a per-release `appcast.xml`.

## What Was Built

**Task 1 — updater wiring + config (commit `14cb643`)**
- `AppDelegate` (already `@MainActor`) owns one `SPUStandardUpdaterController(startingUpdater: true, …)` as a stored property — property initializer runs on MainActor, satisfying Swift 6 strict concurrency.
- A custom `SparkleUpdaterController` `EnvironmentKey` injects the controller into `MenuBarView` and `SettingsView` (mirrors how `appState` is injected). Exactly one construction site in `Sources/`.
- `MenuBarView`: "Check for Updates…" button → `updaterController.checkForUpdates(nil)`.
- `SettingsView`: new "Updates" section with a Toggle bound to a `@State` mirror of `updater.automaticallyChecksForUpdates` (seeded in `.onAppear`, written through on `.onChange`) plus a "Check Now" button.
- `Resources/Info.plist`: added `SUFeedURL`, `SUPublicEDKey` (real public key `nNZTWco7CE8YuNXCK1KwULKwhXHYbkBihX2rJJ02aNk=`), `SUEnableAutomaticChecks`.
- `Tests/SparkleConfigTests.swift`: 2 assertions reading the built `Bundle.main` Info.plist (feed URL exact-match; public key non-empty + base64-shaped). Written first (TDD RED), green after the plist edit.

**Task 2 — release pipeline + docs (commit `4d04d5e`)**
- `release.sh` appends (after staple/checksum, untouched) an appcast block: globs the Sparkle bin in DerivedData, stages only the stapled DMG in an isolated temp dir, runs `generate_appcast` with `--download-url-prefix` for this release's asset path, verifies the output has an `edSignature` and the correct enclosure URL, then `gh release upload … --clobber` (or prints manual instructions). A comment warns that every future release MUST attach `appcast.xml` or auto-updates silently stall.
- `README.md`: new "Auto-updates" section documenting the mechanism, the Keychain-held signing key, and the v1.2.0 caveat.

## Important Note (expected, not a defect)

v1.2.0 shipped **without** this update wiring — it carries no `SUPublicEDKey` and no attached `appcast.xml`. Auto-update benefits begin with the **next** release (the first build to carry both the public key and an attached signed `appcast.xml`). v1.2.0 was not retrofitted.
