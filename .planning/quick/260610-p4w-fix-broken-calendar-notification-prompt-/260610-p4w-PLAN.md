<objective>
Two post-v1.1.0 fixes flagged by the v2.0 milestone audit:

A. **CAL-02 — Repair the dead calendar→notification prompt path.** The calendar prompt is unreachable because `MeetingDetector.handleSignal` routes calendar signals through `DecisionEngine.evaluate`, which requires ≥2 active signals — but the other monitors (audioProcess/micState/windowTitle) are never started in production. A lone `.googleCalendar` signal therefore never fires `onMeetingPrompt`, leaving `NotificationManager.promptToRecord` and the CaddieApp Record/Dismiss handlers dead. Special-case calendar signals so a single active `.googleCalendar` signal fires the prompt directly, with de-dup per event and suppression while a meeting is current. Clear lingering calendar signals on sign-out.

B. **Scope the Google sign-in gate (approved product decision).** ContentView currently blocks ALL UI behind sign-in. Local recordings (list, playback, manual recording) must be accessible signed-out. Show `mainContent` post-onboarding regardless of Google auth; surface the sign-in requirement only in the Today's Schedule area of the sidebar. Onboarding's sign-in requirement (ONB-01) must NOT be weakened.

Purpose: Restore the only working calendar-trigger path (Flow B) and stop locking users out of their local recordings after sign-out.
Output: Working calendar prompt with tests; signed-out users reach local features with a scoped sign-in card.
</objective>
