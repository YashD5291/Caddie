  <action>
    The current actor reentrancy bug: `processNext()` sets `isProcessing = true`, then hits `await updateMeetingStatus(...)`. At this suspension point, Swift actor reentrancy allows another call to `enqueue()` which checks `!isProcessing` -- isProcessing IS true so it doesn't start a new processNext(). BUT: if `enqueue` is called BETWEEN `isProcessing = false` (line 148) and `await processNext()` (line 152), there's a window where both the recursive call and the new enqueue's Task could race.

    Fix approach: Replace the `isProcessing` flag + recursive `processNext()` pattern with a `Task`-based serial queue. Store a single `processingTask: Task<Void, Never>?` that chains continuations:

    ```swift
    private var processingTask: Task<Void, Never>?

    func enqueue(
        meetingId: String,
        database: AppDatabase? = nil,
        onComplete: (@Sendable (String, Result<Void, Error>) -> Void)? = nil
    ) {
        queue.append((meetingId, database, onComplete))
        logger.info("Enqueued meeting \(meetingId) for transcription (queue depth: \(self.queue.count))")

        // Chain onto existing processing task, or start a new one
        let previousTask = processingTask
        processingTask = Task {
            await previousTask?.value  // Wait for previous chain to finish
            await drainQueue()
        }
    }

    private func drainQueue() async {
        while !queue.isEmpty {
            let job = queue.removeFirst()
            await processJob(job)
        }
    }
    ```

    This eliminates the `isProcessing` flag entirely. Each `enqueue` chains a new Task that waits for the previous one to complete, then drains any queued jobs. Since the actor serializes access to `queue` and `processingTask`, this is reentrancy-safe:
    - If `enqueue` is called while `drainQueue` is suspended at an `await`, the actor processes the `enqueue` call, appends to queue, and chains a new Task. The currently-running `drainQueue` will pick up the new item in its `while` loop.
    - Two `drainQueue` calls never run concurrently because each waits for `previousTask?.value`.

    Remove `isProcessing` entirely. Rename `processNext()` to `processJob(_ job:)` (takes a single job, no recursion). Remove the recursive `processNext()` call at the end.

    Update existing tests in TranscriptionPipelineTests.swift to verify:
    1. Add a test that enqueues 2 jobs with a mock engine that introduces an artificial `await Task.yield()` delay, and verify both complete sequentially (first completes before second starts).
    2. Verify no concurrent execution by tracking start/end timestamps in mock engines.
