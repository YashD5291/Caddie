Purpose: UX-01, UX-02, and UX-04 all require information that currently lives deep in AudioRecorder and TranscriptionPipeline but never surfaces. This plan threads that information up through RecordingCoordinator to AppState so Plan 02 can present it.

Output: AppState with `recordingMode` and `pipelineStep` properties, AudioRecorder that reports its mode, TranscriptionPipeline that reports its current step.
</objective>

1. In TranscriptionPipeline, add:
   ```swift
   private var onStepChange: (@Sendable (String, PipelineStep) -> Void)?

   func setOnStepChange(_ callback: (@Sendable (String, PipelineStep) -> Void)?) {
       self.onStepChange = callback
   }
   ```
   The callback takes (meetingId, step).

2. In TranscriptionPipeline.processJob(), call `onStepChange?(meetingId, .step)` before each major step:
   - Before mono mixdown: `onStepChange?(meetingId, .mixdown)`
   - Before ASR: `onStepChange?(meetingId, .transcribing)`
   - Before diarization: `onStepChange?(meetingId, .diarizing)`
   - Before ALAC compression: `onStepChange?(meetingId, .compressing)`
   - After completion (before onComplete): `onStepChange?(meetingId, .idle)`
   - In catch block (before onComplete): `onStepChange?(meetingId, .idle)`

3. In RecordingCoordinator, modify `executeStartRecording` to read `recorder.recordingMode` after `recorder.start()` and pass it up. Change the `onStateChange` callback signature to carry optional metadata:
   ```swift
   private var onStateChange: (@Sendable (RecordingState, RecordingMode?) -> Void)?

   func setOnStateChange(_ callback: (@Sendable (RecordingState, RecordingMode?) -> Void)?) {
       self.onStateChange = callback
   }
   ```
   In the `handle(_:)` method, pass `nil` for mode on non-recording transitions. In `executeStartRecording`, after `recorder.start()` succeeds, call `onStateChange?(.recording(meetingId: meetingId), recorder.recordingMode)`.

   Wait -- simpler approach: keep onStateChange as is but add a separate `onRecordingModeChange` callback:
   ```swift
   private var onRecordingModeChange: (@Sendable (RecordingMode) -> Void)?

   func setOnRecordingModeChange(_ callback: (@Sendable (RecordingMode) -> Void)?) {
       self.onRecordingModeChange = callback
   }
   ```
   Call it in `executeStartRecording` after `recorder.start()`:
   ```swift
   let mode = recorder.recordingMode
   onRecordingModeChange?(mode)
   if mode == .micOnly {
       logger.warning("Recording in mic-only mode -- system audio capture failed")
   }
   ```

4. In RecordingCoordinator.init or a new setup method, wire pipeline step changes:
   After creating the coordinator in AppState.initialize(), wire the pipeline's step callback:
   ```swift
   await pipeline.setOnStepChange { [weak self] meetingId, step in
       Task { @MainActor in
           guard let self else { return }
           self.pipelineStep = step
       }
   }
   ```
   But pipeline is inside coordinator, so coordinator must wire it. Add to RecordingCoordinator:
   ```swift
   private var onPipelineStepChange: (@Sendable (PipelineStep) -> Void)?

   func setOnPipelineStepChange(_ callback: (@Sendable (PipelineStep) -> Void)?) {
       self.onPipelineStepChange = callback
   }
   ```
   In `start()` (or init), wire pipeline's onStepChange to forward through the coordinator's callback:
   ```swift
   await pipeline.setOnStepChange { [weak self] _, step in
       // Use nonisolated(unsafe) or Task to forward
       Task { [weak self] in
           await self?.onPipelineStepChange?(step)
       }
   }
   ```
   Actually, since onPipelineStepChange is @Sendable, just call it directly:
   ```swift
   await pipeline.setOnStepChange { [onPipelineStepChange] _, step in
       onPipelineStepChange?(step)
   }
   ```
   This won't work because the callback can be set later. Simpler: forward in start():

   In RecordingCoordinator.start(), after setting up detector callbacks, wire pipeline:
   ```swift
   nonisolated(unsafe) let stepCallback = onPipelineStepChange
   await pipeline.setOnStepChange { _, step in
       stepCallback?(step)
   }
   ```

5. In AppState.initialize(), after setting onStateChange, wire the two new callbacks:
