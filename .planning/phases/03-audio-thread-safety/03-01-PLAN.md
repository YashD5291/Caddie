    Create Sources/Recording/SPSCRingBuffer.swift -- a single-producer, single-consumer lock-free ring buffer for Int16 audio samples.

    Design constraints (from PITFALLS.md and ARCHITECTURE.md research):
    - Producer is the CoreAudio real-time thread -- MUST NOT allocate, lock, or call Obj-C
    - Consumer is the main thread (flush timer) -- can allocate freely
    - Use OSAtomicFifo or manual atomic indices. Since swift-atomics is a transitive dep but NOT a direct target dependency, do NOT import Atomics. Instead use Swift's native approach:
      - Use `UnsafeMutablePointer<Int16>.allocate(capacity:)` for the backing store
      - Use `UnsafeAtomic<Int>.create(0)` from the stdlib if available in Swift 6, OR use `os_unfair_lock` only for index updates (not the buffer itself), OR simply use `volatile`-equivalent patterns with `UnsafeMutablePointer<Int>` and memory barriers via `atomicMemoryFence`
      - PREFERRED: Use a simple approach -- since SPSC with single writer to writeIndex and single reader to readIndex, use `ManagedBuffer` or just raw pointers with `OSAtomicCompareAndSwap` / Darwin atomics. The simplest correct approach: store head/tail as regular `Int` properties since SPSC guarantees only one thread writes each index. Use `OSMemoryBarrier()` (from Darwin) after writes to ensure visibility.

    Concrete implementation:
    ```swift
    import Darwin

    final class SPSCRingBuffer {
        private let buffer: UnsafeMutablePointer<Int16>
        private let mask: Int  // capacity - 1, for fast modulo
        let capacity: Int
        // Each index is only written by one thread, read by the other
        // OSMemoryBarrier() ensures cross-thread visibility
        private var head: Int = 0  // written by producer, read by consumer
        private var tail: Int = 0  // written by consumer, read by producer

        init(capacity: Int) -- round up to next power of 2
        deinit -- deallocate buffer

        var availableToRead: Int -- (head - tail) using barrier
        var availableToWrite: Int -- capacity - availableToRead

        // Producer (real-time thread): write samples into buffer
        // Returns number of samples actually written (may be less than count if full)
        func write(_ source: UnsafeBufferPointer<Int16>, count: Int) -> Int

        // Consumer (main thread): read samples out of buffer
        // Returns number of samples actually read (may be less than count if not enough available)
        func read(into destination: UnsafeMutablePointer<Int16>, count: Int) -> Int
    }
    ```

    Key implementation details:
    - Capacity MUST be power of 2 so modulo becomes bitwise AND with mask
    - Use `OSMemoryBarrier()` from Darwin after updating head (in write) and tail (in read) to ensure the other thread sees the update
    - write() copies samples one at a time (or use memcpy for contiguous regions) using `buffer[head & mask]`
    - read() copies samples out using `buffer[tail & mask]`
    - No locks, no allocations, no ObjC dispatch in write() -- safe for real-time thread
    - The buffer should handle wrap-around transparently via the mask

    Fix the use-after-free risk in SystemAudioCapture by replacing `Unmanaged.passUnretained(self)` with a separate retained context object.

    The bug (from PITFALLS.md Pitfall 2):
    - Line 287: `Unmanaged.passUnretained(self).toOpaque()` passes self to render callback without retaining
    - Line 396: `Unmanaged<SystemAudioCapture>.fromOpaque(inRefCon).takeUnretainedValue()` recovers self
    - If SystemAudioCapture is deallocated while AudioUnit has an in-flight callback, the callback accesses freed memory

    Fix approach -- use a retained context object that outlives the callback:

    1. Add a private class inside SystemAudioCapture:
    ```swift
    private final class RenderContext {
        var audioUnit: AudioComponentInstance?
        var onBuffer: BufferCallback?
    }
    ```

    2. Add a property: `private var renderContext: RenderContext?`

    3. In setupAudioUnit(), create the context and pass it retained:
    ```swift
    let context = RenderContext()
    context.audioUnit = audioUnit
    context.onBuffer = onBuffer
    self.renderContext = context

    var callbackStruct = AURenderCallbackStruct(
        inputProc: systemAudioRenderCallback,
        inputProcRefCon: Unmanaged.passRetained(context).toOpaque()
    )
    ```

    4. Update the render callback to use RenderContext instead of SystemAudioCapture:
    ```swift
    let context = Unmanaged<SystemAudioCapture.RenderContext>.fromOpaque(inRefCon).takeUnretainedValue()
    guard let unit = context.audioUnit else { return noErr }
    // ... render ...
    context.onBuffer?(bufferPtr, sampleCount)
    ```

    5. In stop(), after AudioOutputUnitStop:
    - Nil out context fields to prevent further callback activity:
      ```swift
      renderContext?.audioUnit = nil
      renderContext?.onBuffer = nil
      ```
    - Release the retained reference:
      ```swift
      if let ctx = renderContext {
          Unmanaged.passUnretained(ctx).release()
          renderContext = nil
      }
      ```

    6. Remove the `fileprivate` access on `audioUnit` and `onBuffer` from SystemAudioCapture since the render callback no longer accesses them directly. Keep them as private. The render callback accesses the RenderContext instead.

    7. Update cleanup() similarly to nil out and release the context.

    IMPORTANT: The RenderContext class does NOT need to be Sendable -- it is accessed from the real-time thread via Unmanaged raw pointer, not through Swift concurrency.

    IMPORTANT: Do NOT change the BufferCallback type or the start() API signature -- AudioRecorder depends on them unchanged.
