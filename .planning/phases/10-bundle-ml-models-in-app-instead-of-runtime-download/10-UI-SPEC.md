### Error Copy by ModelLoadError Case

| Error Case | Copy |
|------------|------|
| bundleNotFound | "Model bundle directory not found in app resources" |
| modelsNotFound | "Model files missing from app bundle: {detail}" |

**Context for error states:** These errors indicate a build or distribution issue, not a user-recoverable problem. The "Retry" button re-attempts loading from the bundle, which may succeed if the issue was transient (e.g., CoreML compilation contention). If it persists, the user should reinstall.

## Interaction Contract

### State Machine (Model Loading Section)

```
[permissions granted] --> Loading --> Ready --> [onboarding complete]
                              |
                              v
                           Error --> [retry] --> Loading
```

Three states, same as current. No new states, no new transitions.

| State | Trigger | Visual | User Action |
|-------|---------|--------|-------------|
| Loading | Permissions granted + Continue tapped | ProgressView + "Loading AI Models" + percentage | Wait |
| Error | loadModelsFromBundle() throws | Error icon + "Model Loading Failed" + error message | Tap "Retry" |
| Ready | modelsReady == true | Checkmark + "Models Ready" | Tap "Get Started" |

### Timing Expectations

| Operation | Current Duration | Expected Duration (Phase 10) |
|-----------|-----------------|------------------------------|
| Model loading | 30s-5min (network-dependent) | 2-10s (local I/O, CoreML init) |
| Progress bar | Slow fill, may stall | Fast fill in two steps (60% ASR, 100% diarization) |
| Timeout | 5 minutes | Removed (D-05) -- local I/O does not need timeout |

**UX implication:** Loading will be fast enough that the progress bar may flash briefly. This is acceptable -- the model-readiness gate (D-07) ensures the user sees confirmation before proceeding.
