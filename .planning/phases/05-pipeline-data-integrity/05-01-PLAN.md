<objective>
Restructure TranscriptionPipeline.processNext() so file deletion is gated on successful completion -- not triggered by defer blocks that run regardless of outcome.

Purpose: Currently the pipeline deletes mono files via `defer` (line 63-66) which runs even if diarization is still reading the file or if the DB write fails. The WAV is deleted at line 121 with `try?` regardless of whether the transcript was actually persisted. This is the primary path to permanent data loss.

Output: A restructured processNext() where:
1. Mono file is deleted explicitly after BOTH ASR and diarization complete (DATA-03)
2. If transcript DB write fails, all source files are preserved (DATA-02)
3. WAV is deleted only after ALAC compression succeeds AND transcript is confirmed in DB AND status is .done (DATA-04)
</objective>

  <behavior>
    RED tests (write these first, all must fail initially):

    1. testMonoFileDeletedAfterASRAndDiarization:
       - Create WAV, enqueue, let pipeline succeed
       - After pipeline completes, verify mono file (caddie_mono_*) does NOT exist in temp dir
       - This confirms mono cleanup still happens on success path

    2. testMonoFilePreservedOnDBWriteFailure:
       - Create WAV, enqueue with a database that will fail on transcript write
       - Use a mock/wrapper approach: after ASR+diarization succeed, make the DB write throw
       - Verify mono file STILL EXISTS after pipeline reports error
       - Verify WAV file STILL EXISTS after pipeline reports error
       - This is the key DATA-02 test

    3. testWAVPreservedOnALACCompressionFailure:
       - Create WAV, enqueue successfully
       - Make ALAC compression fail (e.g., by making output path unwritable)
       - Verify WAV file STILL EXISTS after pipeline reports error
       - Verify meeting status is .error

    4. testWAVDeletedOnlyAfterFullSuccess:
       - Create WAV, enqueue, let pipeline fully succeed
       - Verify WAV file does NOT exist after pipeline completes with .done
       - Verify ALAC file DOES exist

    Note on test approach: For testMonoFilePreservedOnDBWriteFailure, the cleanest approach
    is to add a test hook or use a database that throws on the transcript UPDATE. Since
    AppDatabase wraps GRDB, create the in-memory DB normally but close/invalidate it before
    the transcript write step. Alternatively, inject a `TranscriptionPipelineDelegate` protocol
    that the test can use to intercept. The simplest approach: make a subclass or wrapper of
    AppDatabase that throws on write after N calls. Since AppDatabase is a class, the test
    can track write count.

    Actually -- the simplest approach for testing DB write failure: use a real in-memory DB
    but DROP the meetings table between enqueue and the DB write. This causes the UPDATE to
    throw "no such table: meetings". This is reliable and requires no mocking.
  </behavior>
  <implementation>
    After tests are written and confirmed RED:

    In TranscriptionPipeline.processNext(), restructure the do block:

    1. REMOVE the `defer { try? FileManager.default.removeItem(at: monoURL) }` at line 63-66
    2. After BOTH ASR (step 2) and diarization (step 3) complete, add explicit mono cleanup:
       ```swift
       // Step 4: Clean up mono file -- both ASR and diarization are done with it
       try? FileManager.default.removeItem(at: monoURL)
       ```
    3. Make DB transcript write (step 5) a HARD GATE. If it throws, do NOT proceed to compression:
       ```swift
       // Step 5: Write transcript to DB -- HARD GATE
       // If this fails, preserve ALL source files for retry
       if let db = database {
           try await db.dbWriter.write { dbConn in
               try dbConn.execute(...)
           }
       }
       // Only reach here if DB write succeeded
       ```
    4. Move WAV deletion to AFTER status is confirmed .done:
       ```swift
       // Step 6: Compress WAV to ALAC
       try AudioFileManager.compressToALAC(wavURL: wavURL, outputURL: alacURL)

       // Step 7: Update status to done
       await updateMeetingStatus(meetingId: meetingId, status: .done, database: database)

       // Step 8: Delete WAV only after done status confirmed
       try? FileManager.default.removeItem(at: wavURL)
       ```
    5. In the catch block: do NOT delete mono or WAV -- they must survive for retry.
       The mono file was already cleaned up in step 4 if we got past ASR+diarization.
       If we didn't get past ASR+diarization, mono may or may not exist.
       WAV must ALWAYS survive on error. Currently it does (deletion is in success path).

    Key insight: mono cleanup after step 3 is safe because by that point both consumers
    (ASR step 2, diarization step 3) have already finished reading it. The defer was wrong
    because it ran even if an error occurred during ASR, meaning if ASR threw, defer would
    try to delete a file that may not even exist yet (if createMonoMixdown failed).
  </implementation>
</feature>
